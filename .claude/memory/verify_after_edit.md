# Verify after every edit

**Rule:** After writing or editing any file, **immediately run the relevant verifier and fix
any issues it surfaces** â€” before saying "done" or handing back to the user. Don't leave
warnings or errors for the user to discover.

**Why:** The user repeatedly had to ask for lint cleanup on memory files I'd just edited, and
once had to fix a build error I'd shipped because I didn't rebuild after a change. They
explicitly asked (2026-05-16): *"After writing the code, run the linter/compiler and
automatically resolve any errors or warnings generated by your own changes. Do this for all
documents and files at the end of editing them."*

**How to apply:**

- **`.cs` files** â†’ run `dotnet build CSM.IMTSync.csproj -c Release` after every edit batch.
  Zero warnings, zero errors before reporting done. The post-build target also deploys to both
  game installs, so a clean build also means the test artifact is ready. Build incremental
  failures? Try `rm -rf obj bin && dotnet build` to catch SDK-discovery surprises that hide
  behind cached state.
- **`.md` files** â†’ the IDE diagnostics hook surfaces markdown lint warnings (MD041, MD060,
  MD032, MD033, MD040, MD031) after every Edit/Write. Read the diagnostics block and fix
  anything I introduced before moving on. Pre-existing warnings on lines I didn't touch are
  fair game to leave; warnings on my own added content I must fix.
- **`.csproj` / build config** â†’ run `dotnet build` to verify the project still loads and
  builds. Don't trust that an XML edit is correct just because it parses.
- **Any file** â†’ if the IDE-diagnostics hook (`PostToolUse:Edit hook additional context:
  <ide_diagnostics>`) reports issues that point at lines I touched, fix them in the same turn.

**Prefer auto-fixers when available:**

- Markdown lint: `markdownlint --fix path/to/file.md` will auto-resolve most MD0xx warnings
  (whitespace, blank lines around lists/fences, table-column-style, trailing newlines).
  Run it after a Write/Edit before manual cleanup.
- ESLint (if we ever add JS/TS): `eslint --fix`.
- C# has no comparable auto-fixer for compiler errors â€” fix manually. `dotnet format` can fix
  style issues but won't resolve compile errors.

**Common markdown lint fixes (memorize these patterns):**

- MD041 (no H1 first line) â†’ add an `# H1 Title` line after the YAML frontmatter
- MD060 (table column style) â†’ use `| --- | --- |` not `|---|---|`
- MD032 (lists need blanks) â†’ blank line before AND after every list, even after a bold-prose line
- MD033 (inline HTML from angle brackets) â†’ wrap `<TypeName>` in backticks, or rephrase
- MD040 (fenced code block needs language) â†’ add ` ```text ` or `powershell`/`csharp`/etc.
- MD031 (fenced code needs surrounding blanks) â†’ blank line before and after each ` ``` ` fence

**Exceptions:**

- Pre-existing issues on lines I didn't touch â€” leave them; fixing en-masse is out of scope
  unless the user explicitly asks for a cleanup pass (as they did once).
- The MD041 pattern on memory files: every memory file uses YAML frontmatter + an H1 right
  after. Match this convention rather than restructuring.

**Verify command quick reference:**

```powershell
dotnet build CSM.IMTSync.csproj -c Release
```

A clean build looks like:

```text
Build succeeded.
    0 Warning(s)
    0 Error(s)
```

If either count is non-zero, fix before reporting done.

**Lesson learned 2026-05-16 â€” directory-discovery surprises:**

A clean local clone of [MacSergey/NodeMarkup](https://github.com/MacSergey/NodeMarkup) was
dropped into the working tree at `NodeMarkup-master/` for source-reference reading. The .NET
SDK's `EnableDefaultEmbeddedResourceItems=true` auto-included 40+ `.resx` localization files
from that tree into our csproj, which then triggered `ResGen.exe not supported on .NET Core
MSBuild`. Earlier builds had succeeded because the obj/bin cache hadn't been blown away yet â€”
masking the underlying problem.

**Fix:** added `<DefaultItemExcludes>$(DefaultItemExcludes);NodeMarkup-master\**;bin\**;obj\**</DefaultItemExcludes>`
to the csproj's primary `<PropertyGroup>`. Whenever a new top-level folder appears in the
working tree (third-party clone, scratch space, etc.), check whether SDK item discovery would
sweep it in â€” exclude proactively.
